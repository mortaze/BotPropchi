# Fix: Scheduled Message Keyboard Leakage Between Messages

## Root Cause

**File**: `src/services/scheduled-message.service.ts`, `sendToSingleTarget()` method (line 269)

**The Bug** (line 278-280): The button query filters by `scheduledMessageId` instead of `messageId`:

```typescript
// BUG: fetches ALL buttons for the entire ScheduledMessage
const buttonsFromDb = await prisma.scheduledMessageButton.findMany({
  where: { scheduledMessageId: msg.id },  // ŌåÉ WRONG: should be messageId
  orderBy: [{ row: 'asc' }, { col: 'asc' }],
});
```

**Consequence**: Message 1's keyboard is attached to Message 2 and Message 3 because the same button set is loaded for every iteration.

**Why Auto-Reply doesn't have this bug**: `auto-reply.service.ts:196` correctly filters per-message:
```typescript
const buttonsForMsg = allButtons.filter((b: any) => b.messageId === message.id);
```

## Schema Reference

- `ScheduledMessageButton.messageId` (nullable FK) ŌåÆ which child message the button belongs to
- `ScheduledMessageButton.scheduledMessageId` (required FK) ŌåÆ parent scheduled message
- Repository already has `findButtonsByMessage(messageId)` at `scheduled-message.repository.ts:180`

## Changes

### File: `src/services/scheduled-message.service.ts`

**1. Move button query OUTSIDE the loop (line 276-281)**

Replace the per-iteration DB query with a single query before the loop, using the repository's existing `findButtonsByMessage()`:

```typescript
// BEFORE loop: load ALL buttons for this scheduled message once
const allButtons = await scheduledMessageRepository.findButtonsByScheduledMessage(msg.id);
```

**2. Filter buttons per-message INSIDE the loop (replacing lines 278-288)**

```typescript
// For each message, filter to only its buttons
const buttonsForMsg = allButtons.filter((b: any) => b.messageId === message.id);
const keyboard = buttonsForMsg.length > 0
  ? this.buildInlineKeyboard(buttonsForMsg, message.id)
  : [];
const inlineKeyboard = keyboard.length > 0
  ? { inline_keyboard: keyboard }
  : undefined;
```

Key changes:
- `message.id` (child message ID) instead of `msg.id` (parent scheduled message ID)
- `buttonsForMsg` instead of `buttonsFromDb`
- Each iteration starts with fresh variables ŌĆö no shared state

**3. Add media group guard**

If `message.mediaGroupId` is set, skip keyboard attachment (Telegram doesn't allow reply_markup on media group items):

```typescript
const isMediaGroup = !!message.mediaGroupId;
const finalInlineKeyboard = isMediaGroup ? undefined : inlineKeyboard;
```

Use `finalInlineKeyboard` when building `extra` instead of `inlineKeyboard`.

**4. Update logging (lines 290-317)**

Replace existing keyboard logging with per-message logging that includes:

```
[SchedMsg] KEYBOARD msg=${msg.id} [${i + 1}/${messages.length}] childMsgId=${message.id} buttonCount=${buttonsForMsg.length} hasReplyMarkup=${!!inlineKeyboard} replyMarkupGenerated=${!!inlineKeyboard} buttonSourceMessageId=${message.id}
```

Before the Telegram API call, log the final `reply_markup` with source attribution:

```
[SchedMsg] PRE_SEND msg=${msg.id} [${i + 1}] method=... hasReplyMarkup=${!!finalInlineKeyboard} keyboardSource=childMsg:${message.id} buttons=${buttonsForMsg.length}
```
