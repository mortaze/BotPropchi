# Plan: Auto-Reply Group & Topic Bindings Architecture

## Problem

Current AutoReply model has `targetChatId` (single group) and `targetTopicId` (single topic). The trigger logic in `handleGroupMessage` ignores these fields entirely — any published auto-reply fires in any group. No multi-topic support. The group selection UI is basic (single-select, no topic selection flow).

## Goal

Each auto-reply binds to multiple (GroupId, TopicId) pairs. Only fire when a binding matches. Multi-select topic UI with save/cancel. Full state cleanup.

---

## Step 1: Schema — AutoReplyBinding model

**File**: `prisma/schema.prisma`

Add a new model after `AutoReplySendLog`:

```prisma
model AutoReplyBinding {
  id            Int       @id @default(autoincrement())
  autoReplyId   Int
  chatId        BigInt
  topicId       BigInt?   // null = "all topics" in this group
  isActive      Boolean   @default(true)
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  autoReply     AutoReply @relation(fields: [autoReplyId], references: [id], onDelete: Cascade)

  @@unique([autoReplyId, chatId, topicId])
  @@index([chatId])
  @@index([autoReplyId])
  @@map("auto_reply_bindings")
}
```

Add relation to `AutoReply` model:
```prisma
  bindings      AutoReplyBinding[]
```

**Migration**: `npx prisma db push` (project uses push, not migrate)

## Step 3: Trigger logic — filter by binding

**File**: `src/services/auto-reply.service.ts`

Modify `handleGroupMessage`:
1. Get `chatId` and `topicId` (from `ctx.message.message_thread_id`) from the incoming message
2. Instead of fetching ALL published replies, query bindings: `WHERE chatId = X AND (topicId = Y OR topicId IS NULL) AND autoReply.isPublished = true`
3. Then check keywords only against those filtered replies
4. This ensures isolation — an auto-reply only fires where it has an active binding

Replace `getPublishedWithKeywords()` call with a binding-aware query.

**File**: `src/repositories/auto-reply.repository.ts`

Add `getPublishedForGroup(chatId: bigint, topicId: number | null)` that joins AutoReply ↔ AutoReplyBinding and filters.

## Step 6: Topic multi-select UI (Inline Keyboard)

**File**: `src/bot/handlers/auto-reply.handlers.ts` + `src/bot/keyboards/auto-reply-keyboards.ts`

New action handlers:
- `ar:bind:group:{chatId}` — enter topic selection for group

- `ar:bind:cancel` — discard selections, return to editor

New keyboard builder `buildTopicSelectKeyboard(topics, selectedIds, chatId)`:


Each topic toggle: flips the ID in `pendingTopics[]`, re-renders keyboard in-place (editMessageText).

---

## Step 7: Save/cancel flow

**On save (`ar:bind:save`)**:
1. Read `pendingTopics[]` from state
2. Read `selectedGroup` from state
3. Hard-delete all existing bindings for `(autoReplyId, chatId)`
4. Insert new bindings for each selected topic (or single `topicId=null` for "all topics")
5. Clear selection state
6. Reply with success + show editor

**On cancel (`ar:bind:cancel`)**:
1. Clear selection state
2. Show editor
