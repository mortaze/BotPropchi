## ۱. یافتهٔ فنی کلیدی: رنگ واقعی دکمه در Bot API 9.4 اکنون واقعاً وجود دارد

قبل از هر چیز این را با منبع رسمی تلگرام تأیید کردم (نه حدس): طبق [Changelog رسمی Bot API](https://core.telegram.org/bots/api-changelog#february-9-2026)، در **Bot API 9.4 (۹ فوریهٔ ۲۰۲۶)**:

> «Added the field *style* to the classes KeyboardButton and InlineKeyboardButton, allowing bots to change the color of buttons.»

مقادیر مجاز فیلد `style` دقیقاً سه‌تا هستند (تأیید‌شده از مستندات چند کتابخانهٔ مختلف که همگی این مقادیر را عیناً همین‌طور مستند کرده‌اند):

| مقدار (رشته، انگلیسی، حروف کوچک) | رنگ |
|---|---|
| `"success"` | سبز |
| `"danger"` | قرمز |
| `"primary"` | آبی |
| (فیلد اصلاً ارسال نشود) | رنگ پیش‌فرض همان کلاینت |

نمونهٔ واقعی JSON که تلگرام قبول می‌کند:
```json
{
  "text": "15",
  "callback_data": "news:day:2026-07-15",
  "style": "success"
}
```

### مشکل: کتابخانهٔ `telegraf` هنوز تایپ TypeScript این فیلد را ندارد

این را هم مستقیماً تست کردم: نسخهٔ فعلی و حتی جدیدترین نسخهٔ منتشرشدهٔ `telegraf` (`4.16.3`، که با پکیج `@telegraf/types@7.1.0` کار می‌کند) هنوز `style` را در تایپ `InlineKeyboardButton` تعریف نکرده — یعنی `Markup.button.callback(text, data)` چنین فیلدی را نمی‌شناسد. **این محدودیت فقط در زمان کامپایل TypeScript است، نه در زمان اجرا** — سرور تلگرام این فیلد را کاملاً می‌شناسد و اعمال می‌کند، چون بخشی رسمی از Bot API است؛ کتابخانهٔ کلاینت صرفاً هنوز آپدیت نشده. راه‌حل استاندارد و امن (بخش ۳.۱) یک تابع کمکی کوچک است که فیلد را با یک type assertion کنترل‌شده اضافه می‌کند — نیازی به تغییر نسخهٔ `telegraf` یا هیچ وابستگی جدید نیست.

## ۲. یافته‌های دقیق برای هر باگ (با ارجاع مستقیم به کد فعلی)

### باگ ۲ — رشتهٔ خالی در `ctx.reply()` (علت ریشه‌ای مشترک برای «دکمه لغو نیست» و «reply keyboard عوض نمی‌شه»)

در `src/bot/handlers/news-admin.handlers.ts`، دقیقاً همین ۵ خط مشکل دارند:

```
خط ۷۲:   await ctx.reply('', newsCalendarReplyKeyboard());
خط ۱۴۴:  await ctx.reply('', newsCalendarReplyKeyboard());
خط ۲۲۸:  await ctx.reply('', newsDayEditorReplyKeyboard());
خط ۲۳۳:  await ctx.reply('', newsDayEditorReplyKeyboard());
خط ۲۴۹:  await ctx.reply('', newsCancelKeyboard());
```

هر ۵ مورد، اولین آرگومان (متن پیام) رشتهٔ خالی `''` است. طبق مستندات رسمی [sendMessage](https://core.telegram.org/bots/api#sendmessage)، پارامتر `text` الزامی و غیرقابل‌خالی‌بودن است؛ ارسال متن خالی همیشه با خطای `400 Bad Request` رد می‌شود. یعنی این ۵ پیام **هرگز به تلگرام نمی‌رسند** — دقیقاً به همین دلیل reply keyboard هیچ‌وقت واقعاً عوض نمی‌شود، هرچند کد به‌ظاهر تلاش می‌کند این کار را انجام دهد (حتی کامنت‌های کد مثل `// Reply keyboard for text input (section 6.3)` نشان می‌دهد این تلاش عمدی بوده، فقط پیاده‌سازی‌اش این باگ را داشته).

**یک باگ دومِ به‌هم‌پیوسته هم پیدا شد:** حتی اگر این ۵ خط را فقط با گذاشتن یک متن غیرخالی درست کنیم، مشکل دیگری باقی می‌ماند — هندلرهای `bot.hears('◀️ ماه قبل', ...)`, `bot.hears('ماه بعد ▶️', ...)`, `bot.hears('📍 ماه جاری', ...)`, `bot.hears('◀️ بازگشت به تقویم', ...)` هرکدام با `await ctx.reply(text, kb);` **یک پیام کاملاً جدید** می‌فرستند، نه ادیت پیام موجود. یعنی حتی با اصلاح رشتهٔ خالی، هر بار «ماه بعد» زده شود یک پیام تازهٔ تقویم به چت اضافه می‌شود — دقیقاً برخلاف چیزی که خواستید («ماه بعد و ماه قبل... باید به صورت کامل کار بکنه» و اصل کلی «پیام در حال ادیت شدن باشه»). علتش این است: این هندلرها از callback نیستند (بلکه از `bot.hears` روی متن reply keyboard می‌آیند)، پس `ctx.editMessageText()` مستقیم کار نمی‌کند — نیاز به فرم صریح `ctx.telegram.editMessageText(chatId, messageId, ...)` با یک `messageId` ذخیره‌شده دارند. خوشبختانه `news-state.service.ts` از قبل یک فیلد `messageId` دارد (فقط باید همه‌جا درست پر و استفاده شود — طراحی کامل در بخش ۳.۲).

### باگ ۱ — سکوت کامل ربات برای «📰 اخبار فارکس»

فایل `src/bot/handlers/news-user.handlers.ts` را کامل خواندم. هندلر ورودی:

```ts
bot.hears('📰 اخبار فارکس', async (ctx: any) => {
  const { yesterday, today, tomorrow } = getYesterdayTodayTomorrow();
  const entry = await newsService.getEntry(today);
  const kb = newsUserKeyboard(yesterday, today, tomorrow, today);
  if (entry) { await ctx.reply(...); } else { await ctx.reply(...); }
});
```

**هیچ `try/catch` ندارد.** اگر هر خطایی داخل این تابع رخ دهد (مثلاً یک استثنای دیتابیس، یا هر مشکل دیگری)، کل هندلر throw می‌کند، هیچ پیامی به کاربر نمی‌رسد، و چون هیچ `logger` ای هم در این تابع صدا زده نمی‌شود، اثری در لاگ هم به‌سادگی قابل‌پیدا‌کردن نیست — این دقیقاً منطبق با چیزی است که دیدید: «هیچ چیزی نشون نمی‌ده».

بررسی‌های زیر را هم انجام دادم تا مسیرهای محتمل دیگر را رد یا تأیید کنم:

- **Feature Toggle (`forex_news`) را رد کردم به‌عنوان علت:** `isFeatureEnabled()` در `settings.service.ts` وقتی ردیف دیتابیس وجود نداشته باشد مقدار `true` برمی‌گرداند (`feature?.isEnabled ?? true`)؛ و `ensureDefaults()` این ردیف را با `isEnabled: true` می‌سازد. یعنی اگر این Toggle مسدود می‌کرد، کاربر باید پیام «⛔ این سرویس در حال حاضر غیرفعال است» را می‌دید، نه سکوت کامل. چون سکوت کامل گزارش شده، این مسیر بعید است.
- **نمایش دکمه در منو را هم رد کردم:** منطق `injectServiceButtons`/`buildMainMenuKeyboard` دکمه را تزریق می‌کند مگر این‌که صراحتاً `false` باشد؛ نبودن مقدار هم یعنی نمایش داده می‌شود.
- **باقی‌مانده، محتمل‌ترین توضیح:** یا یک استثنای بی‌صدا داخل خود هندلر (که با نبود try/catch توضیح داده می‌شود)، یا این‌که یک هندلر متنی زودتر-ثبت‌شده (در `registerHandlers`، که قبل از `registerNewsHandlers` در `src/index.ts` اجرا می‌شود) بدون فراخوانی `next()` پیام را می‌بلعد. این دومی را نمی‌شود صرفاً با خواندن کد ۱۰۰٪ تأیید کرد چون `post-handlers.ts` به‌تنهایی ۶ هندلر `bot.on('text')` مجزا و ده‌ها شرط دارد. **به همین دلیل مرحلهٔ صفر لوپ زیر، قبل از هر اصلاحی، این را با لاگ واقعی و اجرای زنده تأیید می‌کند** — حدس‌زدن کورکورانه ریسک باگ جدید دارد.

فارغ از علت دقیق، دو اصلاح زیر **در هر صورت** لازم و بی‌ضرر هستند و بخشی از لوپ‌اند:
1. اضافه‌کردن `try/catch` با پیام fallback به کاربر (یعنی حتی اگر یک علت پیش‌بینی‌نشده دیگر هم وجود داشته باشد، کاربر دیگر هرگز «سکوت کامل» نمی‌بیند).
2. اضافه‌کردن یک خط لاگ در همان ابتدای هندلر (`logger.info('[NewsUser] handler reached...')`) — هم برای تشخیص فوری این مورد، هم برای دیباگ مشابه در آینده.

## ۳. طراحی جدید

### ۳.۱. رنگ واقعی دکمه (جایگزین کامل ⚪️/🟢)

قانون جدید برای شبکهٔ روزهای تقویم:
- برچسب دکمه: **فقط عدد روز**، بدون هیچ ایموجی و بدون کروشه (`15` نه `🟢15` و نه `[15]`).
- اگر آن روز محتوا دارد → `style: 'success'` (سبز) روی خودِ دکمه.
- اگر آن روز خالی است → فیلد `style` اصلاً ست نمی‌شود (رنگ پیش‌فرض).
- نشانگر «امروز» از شبکهٔ تقویم **حذف می‌شود** — چون همان‌طور که سند اول هم مشخص کرده بود، نمایش «امروز» از قبل در **متن پیام** («🗓 امروز: ...») پوشش داده می‌شود؛ نیازی به نشانگر تکراری روی خودِ دکمه نیست و اضافه‌کردنش دقیقاً همان نوع «هک بصری» است که خواستید حذف شود.

پیاده‌سازی امن (چون تایپ `telegraf` این فیلد را نمی‌شناسد — بخش ۱):

```ts
// src/bot/keyboards/news-keyboards.ts
type ButtonStyle = 'primary' | 'success' | 'danger';

function styledCb(text: string, data: string, style?: ButtonStyle) {
  const btn = Markup.button.callback(safe(text), data);
  // @telegraf/types هنوز فیلد style را (نسخهٔ Bot API 9.4، فوریهٔ ۲۰۲۶) تایپ نکرده؛
  // تلگرام خودش این فیلد را کامل می‌شناسد — این فقط یک type assertion امن در زمان کامپایل است.
  return style ? ({ ...btn, style } as typeof btn) : btn;
}
```

و شبکهٔ روز:
```ts
function calendarGridCells(cells: (CellWithContent | null)[]) {
  const rows: ReturnType<typeof styledCb>[][] = [];
  for (let i = 0; i < cells.length; i += 7) {
    const week = cells.slice(i, i + 7);
    rows.push(week.map(cell => {
      if (!cell) return noop(' ');
      return styledCb(String(cell.day), `news:day:${cell.dateKey}`, cell.hasContent ? 'success' : undefined);
    }));
  }
  return rows;
}
```

و متن راهنمای پایین پیام تقویم — خط `🟢 = این روز محتوا دارد   ⚪️ = این روز خالی است` باید **کامل حذف شود** (چون خودِ رنگ دکمه دیگر به توضیح نیاز ندارد؛ نگه‌داشتن این خط بعد از حذف ایموجی از دکمه‌ها هم گمراه‌کننده و هم زائد است).

> پیشنهاد اختیاری (خارج از دامنهٔ اصلی، فقط اگر خواستید): همین مکانیزم `style` را می‌توان برای «🗑 حذف متن» / «✅ تایید حذف» با `style: 'danger'` هم به‌کار برد تا اقدامات مخرب بصراحت قرمز دیده شوند. چون در پیام شما صراحتاً فقط دربارهٔ شبکهٔ تاریخ‌ها گفته شده بود، این را جزو لوپ اصلی **نگذاشتم** — اگر خواستید یک پرامپت جدا برایش می‌نویسم.

### ۳.۲. معماری «یک پیام، همیشه در حال ادیت» + reply keyboard در نقاط انتقال

اصل طراحی: از لحظه‌ای که ادمین «📰 اخبار» را می‌زند تا لحظه‌ای که «🔙 پنل ادمین» را می‌زند، **دقیقاً یک پیام** (همان پیام اول) دائماً ادیت می‌شود — چه محرک آن یک تپ روی دکمهٔ inline (شبکهٔ روزها) باشد، چه یک تپ روی دکمهٔ reply keyboard (ماه بعد/قبل/جاری، افزودن/ویرایش/حذف متن، بازگشت). این یعنی `message_id` این پیام باید **یک‌بار** (در لحظهٔ ورود) در `newsState` ذخیره شود و تا پایان جلسه بدون تغییر باقی بماند.

برای ادیت‌کردن یک پیام از داخل یک هندلر `bot.hears` (که برخلاف `bot.action`، به callback مرتبط نیست)، باید از فرم صریح تلگراف استفاده کرد:

```ts
async function safeEditById(ctx: any, messageId: number, text: string, extra?: any) {
  try {
    await ctx.telegram.editMessageText(ctx.chat.id, messageId, undefined, text, extra);
  } catch (err: any) {
    if (err?.response?.description === 'Bad Request: message is not modified') return;
    throw err;
  }
}
```

**reply keyboard چه زمانی واقعاً باید دوباره فرستاده شود؟** طبق رفتار تلگرام، reply keyboard یک تنظیم سطح-چت است که تا وقتی پیامی صریحاً آن را عوض نکند، ثابت می‌ماند — یعنی ادیت‌کردن متن/inline-keyboard یک پیام (`editMessageText`) هیچ اثری روی reply keyboard پایین صفحه ندارد. پس reply keyboard را فقط لازم است در **نقاط انتقال حالت** دوباره بفرستیم (نه در هر پیمایش):

| از حالت | به حالت | reply keyboard جدید | پیام جدید لازم است؟ |
|---|---|---|---|
| ورود اول (📰 اخبار) | تقویم | ماه‌قبل / ماه‌جاری / ماه‌بعد / پنل‌ادمین | بله (یک پیام کوتاه کنار پیام اصلی) |
| تقویم → تقویم (ماه بعد/قبل/جاری) | تقویم | **بدون تغییر** | خیر — فقط ادیت پیام اصلی |
| تقویم → روز خاص | ویرایشگر روز | (افزودن‌متن) یا (ویرایش‌متن + حذف‌متن) / بازگشت‌به‌تقویم / پنل‌ادمین | بله |
| ویرایشگر روز → تقویم (بازگشت) | تقویم | ماه‌قبل / ماه‌جاری / ماه‌بعد / پنل‌ادمین | بله |
| ویرایشگر روز → حالت دریافت متن | دریافت متن | لغو | بله |
| دریافت متن → ویرایشگر روز (ذخیره یا لغو) | ویرایشگر روز | (بر اساس محتوای جدید) | بله |
| ویرایشگر روز → تأیید حذف | تأیید حذف | تایید‌حذف / انصراف | بله |
| تأیید حذف → ویرایشگر روز | ویرایشگر روز | (بر اساس محتوای جدید) | بله |

هر پیام «فقط برای عوض‌کردن reply keyboard» باید متنی کوتاه و **غیرخالی و بامعنا** داشته باشد (نه فیلر بی‌معنا):

```ts
const KB_LABELS = {
  calendar: '🗓 کنترل‌های تقویم',
  dayEditor: '📋 کنترل‌های این تاریخ',
  awaitingText: 'برای انصراف، «❌ لغو» را در پایین صفحه بزنید.',
  confirmClear: '⚠️ تأیید یا انصراف را در پایین صفحه بزنید.',
};
```

### ۳.۳. کیبورد ویرایشگر روز — وابسته به وضعیت (خالی/پر)، نه ثابت

باگ طراحی دیگری که در `newsDayEditorReplyKeyboard()` فعلی پیدا شد: این تابع **بدون پارامتر** است و همیشه همان دو دکمهٔ ثابت (`➕ افزودن پیام`, `🗑 حذف پیام`) را برمی‌گرداند — یعنی حتی برای روزی که خالی است هم دکمهٔ «حذف» نشان داده می‌شود (حذفِ چیزی که وجود ندارد)، و برای روزی که محتوا دارد دکمهٔ «افزودن» به‌جای «ویرایش» دیده می‌شود. این باید به یک تابع پارامتری تبدیل شود:

```ts
export function newsDayEditorReplyKeyboard(hasContent: boolean) {
  const actionRow = hasContent
    ? ['✏️ ویرایش متن', '🗑 حذف متن']
    : ['➕ افزودن متن'];
  return Markup.keyboard([actionRow, ['◀️ بازگشت به تقویم', '🔙 پنل ادمین']]).resize().persistent();
}
```

(نام دکمه‌ها از «پیام» به «متن» تغییر کرد تا با بقیهٔ این قابلیت — که همه‌جا دربارهٔ «متن» صحبت می‌کند، نه «پیام» — یکدست شود؛ دقیقاً همان یکپارچه‌سازی‌ای که خواستید.)

### ۳.۴. تأیید حذف هم به reply keyboard منتقل می‌شود

مطابق «همگی و همگی باید داخل reply keyboard بیاد»، مرحلهٔ تأیید حذف هم دیگر inline نیست:

```ts
export function newsClearConfirmReplyKeyboard() {
  return Markup.keyboard([['✅ تایید حذف'], ['❌ انصراف']]).resize().persistent();
}
```

با زدن «🗑 حذف متن» (reply keyboard): پیام ویرایشگر روز (همان `messageId` ثابت) به یک متن تأییدیه ادیت می‌شود و reply keyboard به همین کیبورد بالا عوض می‌شود. با زدن «✅ تایید حذف»: حذف واقعی انجام و پیام به حالت «خالی» ادیت می‌شود؛ reply keyboard به `newsDayEditorReplyKeyboard(false)` برمی‌گردد. با زدن «❌ انصراف»: هیچ تغییری در دیتابیس، فقط بازگشت به همان نمایش قبلی.
